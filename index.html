The Problem We Are Solving

Imagine you have a shuffled list of numbers: [3, 6, 8, 5, 2, 7, 1, 4]. Someone asks you — "What is the 4th smallest number?"

The obvious answer is to sort the list first and then pick the 4th element. Sorting gives you [1, 2, 3, 4, 5, 6, 7, 8] — the answer is 4. That works. But here is the problem: you sorted the entire list just to answer one question. If your list has a million numbers, you did a million steps of unnecessary work.

This is called the Selection Problem — given an unsorted list of n elements, find the k-th smallest element as efficiently as possible.

Definition: The k-th smallest element (also called the k-th order statistic) is the element that would appear at position k if the array were sorted. k = 1 means the minimum. k = n means the maximum. k = n/2 means the median.

What is Quickselect?

Quickselect is an algorithm invented by Tony Hoare in 1961 that solves the Selection Problem in expected O(n) time — without sorting the whole array.

It is not a sorting algorithm. It is not a searching algorithm. It is a selection algorithm — its only job is to find one specific rank in an unsorted list and return it.

Why Not Just Sort?

Sorting costs O(n log n) in the best case for comparison-based sorting. That means for 1,000 elements you do roughly 10,000 operations. For 1,000,000 elements you do roughly 20,000,000 operations.

Quickselect on average costs O(n). For 1,000,000 elements, that is roughly 1,000,000 operations — 20 times less work. And you get the exact same answer.

The key insight is this: when you sort, you are ordering every element relative to every other element. But to find one rank, you do not need all of that information. You only need to know: is the answer to the left or to the right of some dividing point?

Real World Uses: Finding the median in a dataset without sorting it. Finding the top-k or bottom-k values. Database engines use selection algorithms to answer ORDER BY LIMIT queries efficiently. Statistics libraries use it for percentile calculations.

The Core Idea: The Pivot

Quickselect works by choosing one element called the pivot and using it to divide the array into two groups:

  • Everything smaller than or equal to the pivot goes to the left
  • Everything greater than the pivot goes to the right
  • The pivot itself is placed in its exact correct sorted position

This process is called partitioning. After one partition, the pivot will never move again — it is in its final position forever.

Key Fact: After partitioning around a pivot that lands at position p, the pivot is guaranteed to be the (p+1)-th smallest element in the array. This is not a guess — it is mathematically certain because every element to its left is smaller and every element to its right is larger.

The Decision After Each Partition

After partitioning, the pivot is at some index p. We compare p to k (the rank we are looking for):

  • If p == k−1: The pivot IS the k-th smallest. We are done. ✅
  • If k−1 < p: The answer is somewhere in the left side. Ignore the right side completely and repeat on A[lo .. p−1].
  • If k−1 > p: The answer is somewhere in the right side. Ignore the left side completely and repeat on A[p+1 .. hi].

This is the crucial step. We throw away at least half the array every time. The discarded half is gone forever — we never look at it again. This is what makes Quickselect fast.

Worked Example: Find the 4th Smallest

Array: [3, 6, 8, 5, 2, 7, 1, 4], k = 4

1
Pick pivot: Last element = 4 (shown in yellow in the visualization)
2
Partition: Scan left to right. 3 ≤ 4 → left. 6 > 4 → skip. 8 > 4 → skip. 5 > 4 → skip. 2 ≤ 4 → left. 7 > 4 → skip. 1 ≤ 4 → left. Place pivot.
3
Result: Array is now [3, 2, 1, 4, 8, 5, 7, 6]. Pivot 4 is at index 3.
4
Decision: We want k = 4, which is index 3 (0-indexed). Pivot landed at index 3. They match — done!
✓
Answer: 4 — the 4th smallest element. Verify: sorted array is [1,2,3,4,5,6,7,8]. Position 4 is indeed 4.

What About the Lomuto Partition?

This tool uses the Lomuto partition scheme. It uses two pointers:

  • i — the boundary between the "small" region and the "large" region. Starts just before the beginning.
  • j — scans from left to right through every element.

Whenever j finds an element smaller than or equal to the pivot, i advances one step and the two elements at i and j are swapped. At the end, the pivot is placed at position i+1. Watch the orange and blue bars in the visualization — those are i and j in action.

What Does Complexity Mean?

Complexity describes how much work an algorithm does as the input gets bigger. We measure it by counting the number of key operations — usually comparisons — and expressing how that count grows with n.

We use Big-O notation. O(n) means the work grows proportionally to n. O(n²) means the work grows with the square of n. O(n log n) sits in between.

Quickselect Complexity

CaseComplexityWhy
Best CaseO(n)Pivot lands exactly at k on the first try. One pass through all n elements and done.
Average CaseO(n)On average, each partition eliminates half the array. Total work: n + n/2 + n/4 + ... = 2n = O(n).
Worst CaseO(n²)Pivot is always the min or max. Each partition only eliminates one element. Total: n + (n−1) + ... + 1 = n²/2.
SpaceO(n)Recursive call stack. Each call uses one stack frame. Worst case n frames deep.
Why is average case O(n)? Each partition scans the active range and eliminates roughly half. So the total work is n (first scan) + n/2 (second scan) + n/4 (third scan) + ... This is a geometric series that sums to 2n — which is O(n).

The Worst Case in Practice

The worst case happens when the pivot is always the smallest or largest element — for example, always picking the last element of an already-sorted array. Every partition puts zero elements on one side and n−1 on the other, so nothing gets discarded efficiently.

How to avoid it: In real software, engineers either shuffle the array randomly before running Quickselect, or pick a random pivot instead of always using the last element. This makes the worst case statistically almost impossible.

What is Divide and Conquer?

Divide and Conquer is a problem-solving strategy with three steps:

  • Divide: Break the problem into smaller subproblems.
  • Conquer: Solve the subproblems (often recursively).
  • Combine: Merge the subproblem solutions back into one answer.

Classic examples include Merge Sort, Quicksort, and Binary Search. Quickselect also uses this paradigm — but with a twist.

How Quickselect Uses D&C

  • Divide: Partition the array around a pivot. The pivot ends up at position p.
  • Conquer: If p == k, done. If k < p, recurse only on the LEFT side. If k > p, recurse only on the RIGHT side.
  • Combine: Nothing to combine. The answer is returned directly. No merging step needed.
The crucial difference from Quicksort: Quicksort recurses on BOTH sides of the pivot because it needs to sort everything. Quickselect only recurses on ONE side because it only needs to find one element. This halves the work at every level — which is why Quickselect is O(n) and Quicksort is O(n log n).

The Recursion Tree: A Straight Line

Look at the Recursion Tree panel above when you run Quickselect. It is a straight vertical line, not a branching tree. Each node shows the range being searched. Each step, the range shrinks. At the bottom is the answer.

Why Greedy Does Not Work Here

A greedy approach would be: scan the whole array, find the minimum, remove it, and repeat k times. This gives you the k-th smallest, but it costs O(n × k) time — which in the worst case is O(n²). Quickselect beats this on average.

Greedy fails here because there is no locally optimal choice that guarantees global optimality without doing at least O(n) work per selection step. The D&C approach is smarter — it narrows the search space geometrically.

Quickselect vs Sort + Access

The most obvious alternative to Quickselect is to sort the array first and then index into it. This always works and is easy to implement. But it costs O(n log n) every time, even if you only need one answer.

Quickselect costs O(n) on average. For a single query on a large dataset, Quickselect wins clearly.

When Sort + Access wins: If you need to answer many different rank queries on the same data, sorting once and indexing many times is cheaper overall. Sort once for O(n log n), then every query is O(1).

Quickselect vs Binary Search

Both algorithms eliminate half the problem at each step. But they are solving different problems:

Why the difference?Binary Search does O(1) work per step (just one comparison). Quickselect does O(n) work per step (scans the whole active range to partition it).

Quickselect vs Quicksort

These two algorithms share the same partition step and the same pivot idea. The difference is entirely in what happens after partitioning:

Quickselect vs Median of Medians

Median of Medians is a more advanced selection algorithm that guarantees a good pivot every time. It always runs in O(n) worst case — unlike Quickselect which has O(n²) worst case.

However, Median of Medians has a much larger constant factor. In practice, randomized Quickselect is faster on average because it avoids the overhead of finding a guaranteed-good pivot. Median of Medians is used when worst-case guarantees are absolutely required.

Summary: Randomized Quickselect is the right choice for most real-world applications. Median of Medians is the right choice when you cannot afford the worst case — for example, in real-time or safety-critical systems.

Things You Must Know

1. Quickselect is a selection algorithm, not a sorting algorithm. It finds one element at a specific rank. It does not sort the array. After Quickselect finishes, the array is partially rearranged but not fully sorted.
2. The pivot always ends up in its correct sorted position. This is the mathematical guarantee that makes the discard safe. After partition, every element to the left is smaller and every element to the right is larger — so the pivot is definitively placed.
3. Discarding is safe, not a guess. When we discard the right side, it is because every element there is larger than the pivot, and the pivot itself is already larger than the rank we need. The answer mathematically cannot be on the discarded side.
4. The worst case is real but avoidable. Always choosing the last element as pivot on a sorted array triggers O(n²). The fix is simple: use a random pivot. This is standard practice in all real implementations.

Common Mistakes

⚠ Mistake 1 — Confusing k indexing: k=1 means the smallest (minimum). k=n means the largest (maximum). Some implementations use 0-indexed k internally — always clarify which convention is being used.
⚠ Mistake 2 — Thinking Quickselect always beats sorting: For a single query, yes. For many queries on the same data, sorting once and indexing repeatedly is cheaper.
⚠ Mistake 3 — Confusing Quickselect with Quicksort: Same partition step, completely different goals. Quicksort sorts everything. Quickselect finds one rank. Quicksort recurses both ways. Quickselect recurses one way.
